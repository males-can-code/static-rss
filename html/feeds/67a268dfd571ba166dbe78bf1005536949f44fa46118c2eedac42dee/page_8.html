Firefox's Move To a Rapid Release Cycle Didn't Just Upset Users
Fri, 18 Jan 2013 15:54:11 GMT

Purportedly, one of the best lessons in software development to have come from the open source movement is the rapid release cycle, and partly because the Google Chrome browser was benefiting from rapid releases, in February of 2011, Mozilla announced that it was moving the Firefox browser to a rapid release cycle. As you can see if you look at the comments on this OStatic post from April of 2011, not every user of Firefox was pleased with the arrangement, due to performance problems that were showing up in the browser.

As it turns out, a worker at Mozilla, Jono Xia, was reportedly just as piqued with the rapid release cycle as many users were. He made this plain in a widely commented on blog post (Xia no longer works at Mozilla). If you're cursing the day that Firefox moved to rapid releases, you may be interested in Xia's view of the situation.

Shortly after Mozilla moved to its rapid release cycle, here is an example comment on Firefox from an OStatic reader:

"The issues are so bad that I am seriously thinking about reverting to 3.6. Using firefox is like trying to control a browser over remote control on a very bad connection."

Jono Xia seemed to be equally miffed in his blog post about Firefox last year:

"Not one person has had anything good to say about the rapid release process. Nearly 100% of my highly unscientific survey volunteered the information -- unasked, unprompted -- that the rapid release process had ruined Firefox for them....I've been thinking a lot about the fundamental disconnect between the developers and the users. I think it comes down to: Software developers have a perverse habit of thinking of updates/new releases as a good thing."

Notably, Xia also singeld out how the Google Chrome team handled things properly:

"Credit where it's due: the way Google handled Chrome updates was very, very smart. They recognized that updates are one of the hardest things to get right, so they solved that problem first, before releasing version 1. The first release of Chrome was little more than an empty box of a browser, but it was wrapped around an excellent updating system. This let them gradually transform that empty box into a full-featured browser, without the users ever realizing they were getting updates."

 I've been using Firefox and Chrome steadily since the new rapid release cycle for Firefox began, and it seems like Mozilla has learned to do updates properly. The performance problems were more of a short-term hiccup.

In a new conversation with Computerworld, Xia clarified some of his comments:

"My point is that software developers and users come from very different perspectives, and that software developers always see an update as a good thing."

Do you wish that Firefox had never been moved to the rapid release cycle?

Design and Enterprise Open Source
Fri, 11 Jan 2013 02:09:56 GMT

Strip away everything you know about your favorite enterprise open source application. Leave everything about it on the table, and ask yourself what the core purpose of the application is. What main function does it perform in your organization? Now, with the answer to that question in your mind, go back to the application and ask a second question: “Why is all this other junk in here?”

A coworker asked me to do a simple task for him the other day. I needed to turn off notifications for a few new servers in Zenoss, so I was going to just set them all into maintenance mode. Except, in the view that held both the list of servers, and a drop down menu that held the menu item I needed there was no way to sort by status. It looked like a regular table, but did not act like one. So, I had to hold down the control key to select multiple servers at once, and change the setting on all of them. Except, the table view forgot my selections after the selections scrolled off the view, so I could only work with items I could see. Except, that the view was so heavy, that it did not scroll smoothly, and I would skip over the servers I was looking for and have to scroll back up. This entire process should be easier.

Aesthetically speaking, Zenoss, as well as many other web apps that I’ve seen, look good. But it is not the looks that are important, it’s the function. The table view mentioned above is only one example of a trend that I’ve seen go on for far too long. Putting a shiny web interface on a task does not necessarily make it better. When evaluating a new application, I want to know two things: “What does it do that is better than what I have now?” and “What do I lose?” You may notice that “How pretty does it look?” is not among the questions that are important. Does it help to make your project look nice? Absolutely, but how it looks should never trump usability.

This concept is why I am still an advocate of plain text and command line management. The command line is not shiny and new, but it can be the most elegant way to manage information, depending on the task. If the option were available (and perhaps it is, I’m not sure) to run a for-loop to put each of the servers into maintenance mode, that surely would have been a more efficient means of interacting with the application than the bloated web interface that had trouble scrolling.

Admittedly, it can be difficult to find HCI experts who are willing to spend their time doing unpaid work on open source projects. Perhaps, instead, those of us who have a vested interest in these enterprise applications can make a concerted effort to help with the user interface design. Reporting bad design as bugs to the maintainers of the projects, and offering solutions where applicable. Good design takes time, but recognizing bad design as a flaw in the software will make the applications we spend so much time in far better.
